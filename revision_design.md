# Problem Statement

Makers with large STL collections live in their OS file browser.

Folders like:

```text
Downloads/stls/
Desktop/3d-prints/
```

become dumping grounds.

Questions like:

- "where is that Raspberry Pi mount?"
- "did I already print this?"
- "which version worked best?"

require opening files one-by-one in a slicer.

Discovery platforms help users find NEW models, but not manage existing collections.

ModelRack is:

> "Plex for 3D print files"

A desktop-native library manager for makers to browse, organize, preview, and eventually track their 3D model collections.

---

# Scope Philosophy

Phase 1 intentionally focuses on:

```text
STL only
```

Specifically:

- binary STL
- ASCII STL

Deferred:

- 3MF
- STEP
- OBJ
- print history
- slicer integration
- database
- AI tagging

The goal is to validate the entire visual pipeline before complexity explodes.

# Constraints

- Desktop-native
- Rust + egui + wgpu
- GPU-first UI
- Dense workshop-style layout
- No document-editor UI patterns
- Solo-developer maintainable
- Smoke-test-first development
- i18n-aware architecture from day one

---

# Premises

1. The pain is real — makers with large STL collections genuinely struggle with OS file browsers

2. Desktop-native is the correct platform — local files, instant response, no uploads

3. Slicer launching is table stakes long-term, but intentionally deferred in Phase 1 to validate browse-and-find workflows first

4. Print history is what eventually transforms ModelRack from a file browser into a true workshop tool

5. Rust + egui + wgpu is the right stack because ModelRack is fundamentally a GPU-first workshop tool, not a document-centric application. The UI should prioritize fast rendering, dense visual layouts, instant interaction, and GPU-accelerated previews over native typography. To avoid common GPU-UI pitfalls, ModelRack must:
   - avoid excessive text wrapping
   - prefer compact metadata
   - truncate long filenames
   - use tooltips for full paths
   - eventually ship bundled CJK-capable fonts

# Architecture Direction

## Chosen Approach

### Approach A — Flat Prototype

The first iteration is intentionally:

- one binary
- one folder
- no database
- no module architecture
- no persistence

The app must:

```text
earn its right to exist visually
before architecture becomes formalized
```

The purpose of Phase 1 is proving:

```text
scan → thumbnail → grid → interaction
```

works smoothly.

---

# Phase 1 Success Criteria

Phase 1 is DONE when:

1. App opens successfully on macOS
2. STL files from a folder appear in a thumbnail grid
3. Scrolling through 100+ models feels smooth
4. Corrupt STL files do not crash the app
5. Empty folders show a proper empty-state message
6. Folder-not-found errors show a clean dialog
7. GPU initialization failure exits cleanly without panic
8. Clicking a model shows metadata instantly

# Revised Technical Strategy

## IMPORTANT: Do NOT Start With Full GPU Rendering

The original design overestimated the need for a complete offscreen wgpu renderer in the first iteration.

That is now intentionally deferred.

The real goal is:

```text
"make the app feel real immediately"
```

—not:

```text
"build a perfect renderer"
```

---

# Revised Rendering Plan

## Phase 1A — Fast Visual Proof

Use:

- placeholder thumbnails
- simple CPU-generated previews
- wireframe/silhouette rendering if needed

Avoid:

- WGSL shaders
- complex render pipelines
- GPU readback optimization
- lighting systems
- PNG encoding

The objective:

```text
see models in a grid
as quickly as possible
```

---

## Phase 1B — Real Thumbnail Renderer

After the grid UI feels good:

- offscreen wgpu rendering
- proper shading
- normalized mesh framing
- thumbnail cache
- async generation workers

This becomes the foundation for:

- future 3D preview viewport
- orbit camera
- real mesh inspection

---

# Thumbnail Pipeline (Revised)

## Initial Pipeline

```text
STL parse
↓
simple preview generation
↓
RGBA buffer
↓
egui::ColorImage
↓
TextureHandle
```

IMPORTANT:

```text
NO PNG ENCODING
```

in Phase 1.

PNG only matters later for:

- disk cache
- export
- persistent thumbnails

Avoid:

```text
GPU texture
↓
PNG encode
↓
decode
↓
GPU upload
```

This is unnecessary overhead.

---

# Async Thumbnail Strategy

Thumbnail generation MUST happen off the UI thread.

Recommended architecture:

```text
UI Thread
↓
placeholder thumbnail

Worker Thread
↓
generate preview

Channel/Event
↓
replace placeholder texture
```

Without this, scrolling large folders will stutter badly.

- a text editor
- a note-taking app
- a word processor

- Blender
- OBS
- PrusaSlicer
- Cura

More than:

- Notion
- Word
- Obsidian

---

# Text Rendering Rules

Because egui uses GPU-rendered text:

## DO

- short labels
- dense cards
- truncation
- tooltips
- compact metadata

## DO NOT

- multiline wrapping everywhere
- long flowing paragraphs
- rich text layouts
- typography-heavy UI

---

# Filename Handling Rules

Long filenames should NEVER wrap aggressively.

Preferred behavior:

```text
truncate + tooltip
```

Example:

```text
raspberry_pi_5_poe_rackmount_v2_f...
```

Hover:

```text
/full/path/to/raspberry_pi_5_poe_rackmount_v2_final.stl
```

---

# i18n Strategy

Phase 1 ships:

```text
English only
```

But the architecture should remain i18n-aware.

Deferred:

- Fluent
- ICU
- translation files
- pluralization systems
- bundled CJK fonts

Phase 1 should simply use:

```rust
const APP_TITLE: &str = "ModelRack";
```

style constants.

---

# Performance Strategy

The first target is NOT:

```text
100k STL files
```

That is future optimization territory.

Phase 1 target:

```text
100–1000 files smoothly
```

This dramatically simplifies implementation.

---

# Virtualized Grid Scope

The original design overstated the complexity of virtualization.

Phase 1 can safely use:

- egui ScrollArea
- chunked loading
- lazy thumbnail generation

Full virtualization can come later if needed.

---

# Folder Scanning Rules

Phase 1 scanner requirements:

- recursive scan
- STL only
- skip corrupt files
- continue scanning on errors
- ASCII STL file size cap:
  - 100MB
